# Elder Care Assistant - Claude Agent Instructions

This file guides AI assistants (Claude) when working with the Elder Care Assistant project.

## Project Overview

**Elder Care Assistant** is an AI-powered health monitoring and companion system for elderly care. It's a FastAPI backend serving an AI-powered chat interface integrated with health tracking, medication reminders, family communication, and emergency response features.

- **Type**: Backend API (Python/FastAPI)
- **Primary User**: Elderly individuals and their families
- **Key Integration**: Claude API for conversational AI
- **Database**: PostgreSQL with Redis caching

## Project Priorities

1. **User Safety First**: Never compromise on security, especially for health data
2. **Privacy**: Elderly users' health information is highly sensitive
3. **Reliability**: Health monitoring features must be dependable
4. **Accessibility**: Interface should work for users with varying technical literacy
5. **Family Connectivity**: Enable meaningful communication between elderly and families

## Architecture Overview

### Core Layers

- **API Layer** (`app/api/routes/`): FastAPI endpoints for all features
- **Service Layer** (`app/services/`): Business logic and integration logic
- **Data Layer** (`app/models/`, `app/database/`): SQLAlchemy ORM and database operations
- **Middleware** (`app/middleware/`): Auth, error handling, CORS

### Key Dependencies

```
FastAPI → Pydantic → SQLAlchemy → PostgreSQL
                   → Redis
                   → Claude API
```

## Development Patterns

### Code Style

- **PEP 8 compliance** with max line length of 100 chars
- **Type hints** required on all functions
- **Docstrings** for public functions (single-line preferred)
- **Snake_case** for functions/variables, **PascalCase** for classes

### Database Access Pattern

Always use SQLAlchemy session:
```python
from app.database.db import get_db

async def get_user_health(user_id: int, db: Session):
    return db.query(HealthRecord).filter(
        HealthRecord.user_id == user_id
    ).order_by(HealthRecord.created_at.desc()).first()
```

### Authentication Pattern

Use JWT tokens:
```python
from app.middleware.auth import get_current_user
from app.schemas import User

async def health_endpoint(current_user: User = Depends(get_current_user)):
    # current_user is authenticated
    return {"user_id": current_user.id}
```

### Error Handling

Use FastAPI HTTPException:
```python
from fastapi import HTTPException

raise HTTPException(status_code=404, detail="User not found")
raise HTTPException(status_code=400, detail="Invalid health data")
```

### Async Patterns

All endpoints should be async:
```python
@router.get("/data")
async def get_data():
    # Use await for I/O operations
    result = await some_async_operation()
    return result
```

## Important Configuration Points

### Environment Variables

Key variables in `config.py`:
- `CLAUDE_API_KEY`: Critical for chat functionality
- `DATABASE_URL`: PostgreSQL connection string
- `REDIS_URL`: Cache/session store
- `JWT_SECRET`: Must be strong in production
- `SMTP_*`: Email notifications (optional)
- `TWILIO_*`: SMS notifications (optional)

### Security Notes

- `REQUIRE_EMAIL_VERIFICATION`: Set to `true` in production
- `REQUIRE_2FA`: Consider `true` for healthcare context
- `ALLOW_REGISTRATION`: Toggle to restrict new users
- `CORS_ORIGINS`: Must be explicitly listed (not `*`)
- `RATE_LIMIT_*`: Adjust based on expected usage

## Common Tasks

### Adding a New Endpoint

1. Create schema in `app/schemas/schemas.py`
2. Create model if needed in `app/models/models.py`
3. Add business logic to `app/services/services.py`
4. Create route in `app/api/routes/`
5. Include router in `app/main.py`

### Modifying Database Schema

1. Update model in `app/models/models.py`
2. Create migration (if using Alembic): `alembic revision --autogenerate`
3. Update schemas if input/output changes
4. Test thoroughly (production migrations are risky)

### Integrating Claude API Updates

1. Update `CLAUDE_MODEL` in `config.py`
2. Review model differences/deprecations
3. Update any prompt templates if needed
4. Test chat functionality thoroughly
5. Update version in `main.py` if major change

## Known Considerations

### Healthcare Context

- All health data is PII (Personally Identifiable Information)
- Regulatory compliance may be needed (HIPAA in US, GDPR in EU)
- Never cache sensitive health data insecurely
- Audit logging for health data access
- Patient consent for data sharing

### Elderly Users

- May have lower technical literacy
- Accessibility is critical (large fonts, high contrast, simple navigation)
- Reliability and uptime are essential
- Clear error messages (not technical jargon)
- Family members may be helping with usage

### AI Chat Safety

- Claude can provide information but not replace medical advice
- Should acknowledge when advice is not professional medical guidance
- Track and audit sensitive conversations
- May need human review for certain responses
